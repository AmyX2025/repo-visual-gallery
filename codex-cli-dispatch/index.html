Codex CLI Dispatch 到底是什么东西？

简单来说，它是一个"任务分发器"——你把编程任务丢给它，它就在后台悄悄让 OpenAI 的 Codex 帮你写代码， 写完了自动通过 Telegram 通知你，整个过程不需要你盯着屏幕等。就像你把脏衣服扔进洗衣机，设好程序就可以去做别的事了。

codex-cli-dispatch/ ├── SKILL.md <- 技能说明书，告诉 AI 助手怎么使用这个工具 ├── README.md <- 英文使用手册 ├── README_CN.md <- 中文使用手册 ├── scripts/ │ └── dispatch.sh <- 核心脚本，整个项目的"大脑"，负责分发任务 ├── LICENSE <- 开源许可证，MIT 协议（随便用） └── .gitignore <- 告诉版本管理工具哪些文件不用跟踪
1

架构层次

项目的结构像一栋楼，从上往下看，每一层都有各自的职责。点击任意一层查看详情。

1 🧑 用户 / AI 助手层 发起任务的人或 AI，就像餐厅里点菜的顾客
OpenClaw Agent 终端用户
这是最顶层——任务的发起者。可以是你自己在终端里手动输入命令，也可以是你的 AI 助手（比如 OpenClaw 里的小龙虾）自动调用。 就像在快餐店点单：你说要什么，后面的厨房就开始忙活。你只需要用 --prompt 告诉它"要做什么"， 剩下的事情它全帮你搞定。
2 📦 任务分发层（dispatch.sh） 核心调度员，收到订单后安排一切，就像餐厅经理
dispatch.sh SKILL.md
这是整个项目的"大脑"，一个 211 行的 Bash 脚本。它的工作分 7 步： 解析你的命令参数、记下任务信息、确保工作目录有版本管理、拼装 Codex 命令、启动 Codex 干活、保存结果、最后通知你。 就像一位尽职的餐厅经理：接单、记录、备料、通知厨师开始做、盯着出餐、最后叫服务员上菜。
3 🤖 AI 编程引擎层（Codex CLI） 真正写代码的 AI，就像厨房里做菜的大厨
codex exec gpt-5.3-codex
这层是 OpenAI 出品的 Codex CLI，一个能在终端里运行的 AI 编程助手。它接收任务指令后， 会自动阅读你的代码、理解上下文、然后像一位资深程序员一样写出新代码或修复问题。 它在"沙盒"（一个隔离的安全空间）里工作，不会乱动你不让它碰的文件。 就像大厨只在自己的工作台上操作，不会跑到收银台去乱按。
4 💾 数据存储层 保存任务记录和输出结果，就像餐厅的订单记录本
latest.json task-meta.json task-output.txt
每次任务完成后，结果会被整齐地保存在三个文件里：task-meta.json 记录"谁在什么时候派了什么任务"， task-output.txt 保存 AI 写代码时的完整输出，latest.json 是一份精简的结果摘要。 就像餐厅每天的流水账：什么时候接了什么单、做了多久、最后出了什么菜。
5 📢 通知和唤醒层 任务完成后通知你，还能叫醒 AI 助手继续干活
Telegram Webhook OpenClaw Hooks
这层负责"通知到位"。任务做完后，它会做两件事： 第一，通过 Telegram 给你发一条消息，告诉你"活干完了，结果如何"； 第二，通过一个叫 Webhook 的机制（就像设了一个闹钟，事件发生了自动响）去叫醒你的 AI 助手， 让它知道可以进行下一步了。这样你完全不用盯着屏幕等——做完了它自己会来找你。
6 ☁ 外部服务层 项目依赖的外部工具和平台
OpenAI API Telegram Bot API Git
项目本身很轻量，但它依赖几个外部服务来运转： OpenAI 的云端 AI 服务是"大厨的技能来源"，Telegram 的机器人接口是"送信的邮差"， Git 版本管理是"记录每次修改的时光机"。这些服务就像餐厅的水电供应—— 厨房再好，没水没电也做不出菜。

工作流程

一个任务从发出到完成，经历这 7 个阶段。就像一条流水线，每个环节各司其职。

1
解析参数

读取你输入的命令参数：要做什么任务、在哪个文件夹、用什么 AI 模型、安全等级多高。

→
2
记录元数据

把任务信息写进 task-meta.json，相当于在订单本上登记："几点几分，谁派了什么活"。

→
3
初始化 Git

检查工作目录是否有版本管理。没有的话自动创建，确保 Codex 能正常工作。

→
4
拼装命令

根据你的参数，拼出完整的 codex exec 命令——模型、沙盒模式、图片附件等全部安排好。

→
5
执行 Codex

启动 Codex CLI，让 AI 大厨正式开始写代码。输出同时保存到文件里。

→
6
保存结果

AI 干完活后，更新元数据状态，把精简结果写进 latest.json，方便后续查看。

→
7
通知 + 唤醒

给 Telegram 发消息通知你，同时通过 Webhook 唤醒你的 AI 助手，让它继续处理后续任务。

术语词典

页面上出现的技术术语，全部用大白话解释一遍。

Codex CLI

OpenAI 出品的"终端编程机器人"。你在命令行里告诉它要做什么，它就自动帮你写代码、修 bug、做重构。就像请了一位随叫随到的程序员，24 小时待命。

OpenClaw

一个开源的 AI 助手管理平台，帮你把各种 AI 工具整合在一起。可以理解为"AI 助手的管家"，管着你的 AI 能用什么工具、怎么跟你沟通、记住什么。

dispatch（分发）

把任务交给别人去做。就像快递公司的"调度中心"——收到包裹后分配给哪个快递员去送。这里是把编程任务分配给 Codex 去执行。

沙盒（Sandbox）

给程序画一个"安全围栏"。在围栏里面，程序可以自由活动；但出了围栏就不行。这样即使程序出了问题，也不会影响围栏外面的东西。有三个等级：只读（只能看）、工作区写入（能改指定文件夹）、完全访问（什么都能动，要小心）。

nohup

让程序在"后台默默干活"的命令。平时你关掉终端窗口，正在运行的程序也会跟着停。加了 nohup 就相当于告诉它"我走了你也别停，干完再说"。

Bash

电脑的"命令行语言"。你在黑底白字的终端窗口里敲的命令，大部分就是 Bash 语言。这个项目的核心脚本就是用 Bash 写的，就像用中文写了一份操作步骤。

JSON

一种通用的数据记录格式，用花括号和冒号来组织信息。就像填表格——"姓名: 张三, 年龄: 25"。程序之间交换数据时经常用这种格式，因为人能看懂、机器也能处理。

jq

专门处理 JSON 数据的小工具。如果 JSON 是一份表格，jq 就是帮你查表、改表、生成新表的"数据助手"。在这个项目里用来生成和读取任务信息文件。

Git

代码的"时光机"。每次修改代码后，Git 会帮你拍一张"快照"。出了问题可以随时穿越回之前的版本。几乎所有程序员都用它来管理代码。

git init

在一个文件夹里"启用时光机"。运行这个命令后，Git 就开始跟踪这个文件夹里的所有变化。dispatch.sh 会自动帮你执行这一步。

Webhook

给系统设一个"自动通知机制"——当某件事发生时（比如任务完成了），自动发一条消息给指定地址。就像设了一个闹钟，到点了自动响，不用你一直盯着看。

API

两个软件之间"打电话的接线员"。软件 A 想用软件 B 的功能，就通过 API 打个"电话"，说明要什么，B 就把结果传回来。这个项目通过 API 调用 OpenAI 的 AI 能力和 Telegram 的消息服务。

退出码（Exit Code）

程序结束时留下的一个数字"成绩单"。0 表示"一切正常，任务完成"；非 0（比如 1）表示"出了问题"。dispatch.sh 根据这个数字判断 Codex 的任务是成功还是失败。

Telegram Bot

Telegram 里的"自动回复小助手"。你可以创建一个虚拟机器人账号，让它自动发消息、接收命令。这里用它来自动推送任务完成的通知到你的群组。

PIPESTATUS

Bash 里的一个特殊变量，记录"流水线"上每一步的成功或失败状态。dispatch.sh 用它来获取 Codex CLI 的真实退出码，即使输出被同时保存到了文件。

技能插件（Skill）

给 AI 助手"学习新本领"。就像手机装 App 一样，给 AI 装一个技能插件，它就多会一项本事。codex-cli-dispatch 就是一个技能，让 AI 助手学会"分发编程任务"。

full-auto（全自动模式）

让 Codex 不需要你的确认就自动执行所有操作。平时 Codex 每写一段代码都会问你"可以吗？"，开了全自动就变成"闷头干，干完再说"。在沙盒保护下这样做是安全的。

curl

命令行里的"浏览器"——可以向网络地址发送请求、获取数据，只是没有图形界面。dispatch.sh 用它来给 OpenClaw 发唤醒信号，就像用对讲机呼叫同事。
