Chapter 01

为什么需要 OpenClaw？
从云端到本地的觉醒

想象一下，你的 AI 助手就像一个住在别人家的管家——每次你需要它，它都要打长途电话回总部请示。

2024 年末，当 ChatGPT、Claude、Gemini 等云端 AI 助手成为主流时，大多数人已经习惯了这种模式：你在浏览器或 App 里输入问题，AI 在远程服务器上思考，然后把答案传回来。看起来很方便，对吧？

但这背后隐藏着三个问题：

问题一：隐私的代价

你和 AI 的每一次对话，都会被上传到云端服务器。无论是工作中的敏感数据、个人隐私，还是你的创意想法，都要经过第三方的手。你无法确定这些数据会被如何处理、存储多久、是否会被用于训练模型。

问题二：网络的束缚

云端 AI 依赖网络连接。没有网络？AI 就失效了。网络慢？体验就会卡顿。想在飞机上、地下室、或者网络受限的环境中使用？抱歉，不行。

问题三：平台的围墙

大多数云端 AI 都有自己的专属界面：ChatGPT 有网页版，Claude 有自己的 App。如果你想在 WhatsApp、Telegram、Slack、Discord 等你日常使用的平台上直接和 AI 对话，往往需要复杂的配置，或者根本做不到。

OpenClaw 的诞生，就是为了打破这些限制。

它的创始人 Peter Steinberger (@steipete) 在 2024 年 11 月开始构建这个项目时，有一个清晰的愿景："AI 助手应该跑在你自己的设备上，集成到你已经在用的平台里，遵循你自己的规则。"

这不是一个"更好的 ChatGPT 替代品"，而是一个完全不同的范式——本地优先、渠道无关、可完全掌控。

Chapter 02

OpenClaw 的核心哲学
不只是工具,而是你的数字伙伴

如果 AI 助手是一个管家，OpenClaw 的哲学是：让它住在你家，而不是住在别人的酒店。

OpenClaw 的设计哲学可以用四个核心原则概括，这些原则贯穿了整个项目的工程决策。

原则一：本地优先，始终在线

OpenClaw 不是一个"需要时才启动"的工具，而是一个常驻的后台程序（守护进程，daemon）——就像你电脑上的杀毒软件或输入法，开机就默默启动，一直在后台等着，不需要你手动打开。

在 macOS 上，它通过 launchd 启动（系统开机时自动管理后台程序的机制）；在 Linux 上，通过 systemd 管理（同样是 Linux 系统的开机自动启动管理器）。你不需要每次使用前打开一个 App 或网页——它就在那里，等着你。

"OpenClaw 是那种你会忘记它存在的工具，直到你需要它的时候，它总是在。"

原则二：渠道无关——在你已有的平台上工作

大多数 AI 助手都会强迫你使用它们的专属界面。但 OpenClaw 的理念是：在用户已经在用的地方出现。

你习惯用 WhatsApp 和朋友聊天？那就直接在 WhatsApp 里和 AI 对话。
你的团队在 Slack 上协作？那就让 AI 也加入 Slack。
你喜欢用 Telegram 的机器人功能？OpenClaw 支持。

OpenClaw 支持 20+ 个消息平台，包括 WhatsApp、Telegram、Slack、Discord、Signal、iMessage、Matrix、IRC、Teams、LINE 等等。这不是简单的"聊天机器人"集成，而是深度的多渠道统一。

原则三：可黑客化——TypeScript 让一切可修改

OpenClaw 选择 TypeScript 作为主要开发语言，这是一个故意的工程决策。创始人在 VISION.md 文档中明确写道：

"OpenClaw 主要是一个编排系统：提示词、工具、协议、集成。我们选择 TypeScript 是为了让 OpenClaw 默认就是可黑客的。它被广泛使用、迭代快速、易于阅读、修改和扩展。"

相比 Python 或 Rust，TypeScript 的优势在于：
• 易读性：非专业开发者也能看懂大部分代码
• 生态丰富：npm 生态系统有海量的库可以直接使用
• 类型安全：TypeScript 的类型系统能避免很多运行时错误
• 社区广泛：几乎所有前端开发者都熟悉 TypeScript

这意味着，如果你想修改 OpenClaw 的行为、添加自定义功能、或者调试问题，门槛非常低。你不需要成为系统编程专家——只需要会一点 JavaScript/TypeScript 就够了。

原则四：安全但不失能力——平衡的艺术

这是 OpenClaw 最微妙的设计权衡。大多数本地 AI 工具要么完全开放（任何人都能访问），要么完全封闭（只能在沙盒里运行）。OpenClaw 选择了中间路线：

默认安全，但保留能力。

比如：
• 默认情况下，只有配对（paired）的用户才能通过 DM 访问 AI
• 但你可以选择开放访问（比如在团队内部使用）
• 提供可选的沙盒隔离（Docker），但不是强制的
• 有 Owner-only 命令和提权模式，确保敏感操作需要授权

这种设计哲学被总结为："让风险路径显式化，由操作者控制。"不是替用户做决定，而是给用户选择的权力。

Chapter 03

架构全景：Gateway 中心的交响乐
为什么需要一个"总调度中心"？

想象一座繁忙的国际机场：数百架飞机来自不同航空公司，降落在不同跑道，但都需要一个调度塔来协调。OpenClaw 的 Gateway 就是这样的角色。

当你在 WhatsApp 上给 AI 发消息、在 Slack 上 @ 它、或者在 Telegram 群组里提问时，所有这些请求都会汇聚到一个地方：Gateway。

Gateway 是 OpenClaw 架构的核心，它是一个长期运行的 WebSocket 服务器，默认监听在 127.0.0.1:18789。它的职责不是"处理 AI 请求"（那是 Agent Runtime 的工作），而是编排和路由。

为什么不是"点对点"架构？

你可能会问：为什么不让每个消息平台直接连接到 AI 引擎？为什么需要一个中间层？

这就是 OpenClaw 的第一个工程化突破。让我用一个类比来解释：

反模式：点对点的混乱

假设你要举办一个派对，邀请了 20 个朋友。如果你让每个朋友直接和其他 19 个人协调时间、地点、带什么食物，那会是什么情况？N×N 的复杂度——每个人都要和其他所有人沟通。

在技术系统中，这意味着：
• WhatsApp 插件需要知道如何和 Agent 通信
• Telegram 插件需要知道如何和 Agent 通信
• Slack 插件需要知道如何和 Agent 通信
• ...重复 20 次

每个插件都要实现相同的逻辑：会话管理、消息队列、错误处理、重试机制。这是代码重复的灾难。

OpenClaw 的方案：中心化协调

Gateway 的设计就像派对的主办者：所有人只需要和你一个人沟通。你负责协调时间、预订场地、分配任务。

在 OpenClaw 中：
• 所有消息平台（WhatsApp、Telegram、Slack...）只需要和 Gateway 对话
• Gateway 负责路由消息到正确的 Agent
• Gateway 管理所有的会话状态、消息队列、事件订阅
• Agent 只专注于"思考和回复"，不关心消息来自哪个平台

这种设计的好处是：
1. 关注点分离：每个组件只做一件事，做好一件事
2. 可扩展性：添加新平台只需要实现一个"Gateway 适配器"
3. 统一的控制平面：所有客户端（CLI、macOS App、Web UI）都通过同一个协议连接

Gateway 的四大职责

作为"总调度中心"，Gateway 具体做什么？

1. 消息路由（Routing）

当一条消息从 WhatsApp 进来时，Gateway 需要决定：这条消息应该发给哪个 Agent？是主 Agent？还是某个特定的子 Agent？

OpenClaw 支持多智能体路由（我们会在第五章详细讲）：同一个 Gateway 可以托管多个隔离的 Agent，每个 Agent 有自己的工作空间、配置、会话。Gateway 通过优先级规则决定路由目标。

2. 会话管理（Session Management）

Gateway 是所有会话状态的唯一真实来源（single source of truth）。它负责：
• 创建新会话
• 跟踪会话状态（活跃、空闲、已关闭）
• 将多个渠道的对话合并到同一个会话（例如：你在手机和电脑上用不同账号，但 AI 知道是同一个人）
• 持久化会话记录到 JSONL 文件

3. 事件系统（Event System）

想象一个电台广播：电台只管发信号，不在乎有多少台收音机在收听；收音机自己调到对应的频道，想听就听，不想听就不管。

Gateway 的事件系统就是这个逻辑。Gateway 不停地对外"广播"各种状态信号：
• AI 开始思考了→ 发出信号
• 新消息进来了→ 发出信号
• 任务完成了→ 发出信号

各个部件（比如 macOS App 上的状态图标）自己"调频"监听它关心的信号——收到"AI 在思考"，就显示一个转圈动画。它们之间不需要直接打招呼，各自独立工作。这就是事件总线（发布-订阅模式，Pub-Sub）——让系统各部分解耦，互不依赖。

4. 工具和插件管理（Tool & Plugin Registry）

Gateway 还负责加载和管理所有的工具（Tools）和插件（Plugins）。比如浏览器控制、Canvas 渲染、Cron 任务、文件操作等能力，都是通过 Gateway 注册和分发的。

为什么这是工程化突破？

Gateway 中心化架构的价值，在于它解决了复杂系统的可组合性问题。

对比其他 AI 助手项目：
• 大多数开源 AI 助手要么是单体应用（monolith），要么是"一个平台一个机器人"的模式
• 如果你想同时支持多个平台，往往需要运行多个独立的进程，每个进程有自己的状态管理
• 想要在不同平台之间共享会话？基本不可能，或者需要复杂的外部数据库同步

OpenClaw 的优势，用直白的话来说：

一：单一进程管理所有渠道
想象你雇了一个私人助理。其他 AI 助手的做法是：WhatsApp 一个助理，Telegram 一个助理，Slack 又一个——他们互不相识，各记各的笔记。你跟 WhatsApp 的助理说"帮我记住会议在周五"，Telegram 那边的助理完全不知道这件事。OpenClaw 的做法是：只有一个助理，坐在总控台前，不管你从哪个 App 发消息，他都是同一个人，记着同一本笔记。

二：统一的会话模型，跨平台对话连续性
既然只有一个助理，那他的"笔记本"当然也只有一本。你上午在 WhatsApp 上聊了工作安排，下午切到 Telegram 继续问进展，AI 不会一脸茫然地说"你是谁？"——因为所有渠道的对话都汇入同一个会话。这是一个非常重要的设计，我们会在下一章专门展开讲。

三：插件式扩展，添加新平台只需几百行代码
OpenClaw 把每个平台的接入设计成一个"插头"。Gateway 是固定的插座，WhatsApp 是一个插头，Telegram 是另一个插头。你想加微信或飞书？有人写一个"微信插头"，插进去就能用，完全不用动 OpenClaw 的核心代码。这就是为什么 OpenClaw 能支持 20 多个平台——不是团队一个个手工做的，而是任何人都可以做一个新插头接进来。

四：类型安全的协议
当系统里有这么多模块互相传消息，怎么防止"传话传错了都不知道"？想象医院的化验单：不管哪个科室开单，格式都一样——病人姓名、项目代码、数值、单位，每一格规定好，填错了系统立刻报警，这张单子不会流转下去。OpenClaw 内部各模块传递信息时，采用的就是这套标准化格式——任何错误立即被发现，不会悄悄流传造成难以排查的问题。

实际运行：Gateway 的生命周期

当你启动 OpenClaw 时，Gateway 会：

步骤 1：初始化
• 加载配置文件 ~/.openclaw/config.yaml
• 启动 WebSocket 服务器
• 加载所有 Channel Providers（WhatsApp、Telegram...）
• 初始化 Agent Runtime

步骤 2：监听连接
• 等待客户端连接（CLI、macOS App、WebChat）
• 验证连接的权限（token 认证）
• 建立长连接，准备接收控制命令

步骤 3：处理消息流
• Channel Provider 接收到消息（比如你在 WhatsApp 上发了一条消息）
• 消息被规范化为统一的内部格式
• Gateway 路由消息到目标 Agent
• Agent 处理并返回回复
• Gateway 将回复发送回原始渠道

步骤 4：守护进程模式
• Gateway 持续运行，不会因为没有活动而退出
• 通过 systemd（Linux）或 launchd（macOS）管理，系统启动时自动运行
• 崩溃时自动重启

🔧 工程化突破亮点

问题：如何让 20+ 个消息平台共享同一个 AI 引擎，而不产生 N×N 的代码重复？

价值：清晰的关注点分离、可插拔的扩展性、跨平台的会话连续性、类型安全的协议。

3. Gateway 的四大职责：消息路由、会话管理、事件系统、工具注册

等等——这好像也没什么技术难度？

读完上面这些，你可能会有一个感受：「把消息汇到一个地方」这个想法确实不难，任何程序员都能想到。OpenClaw 的突破好像也没什么了不起的？

这个感受基本对，但需要补充一个细节。

OpenClaw 真正的工程价值，在于它做了一系列正确的决策：选择"本地运行"而非云端服务（意味着你的数据不经过任何公司的服务器）；设计出一套真正能让别人写插件的架构，而非只有自己能维护的私有系统；把 session 管理、安全模型、多 Agent 路由这些"看不见的细节"做对。这些选择本身不是高难度的技术发明，但做错了就会崩，做对了才能让 20 多个渠道、多个 Agent、多种设备都稳定运行在你的本地电脑上。

打个比方：盖一间房子的原理不难，但把地基打稳、把水电设计好，需要的是经验和判断力，不是什么高精尖技术。

会话管理的艺术
让 AI 记住你的每一次对话

想象医生的病历本：无论你从哪个诊所来、用哪个电话预约，医生都能翻开病历，看到你的完整就诊历史。OpenClaw 的会话管理就是这样。

当你在 WhatsApp 上和 AI 聊了半小时，然后切换到电脑上的 Telegram 继续对话，你期望 AI 能记住之前的上下文吗？当然应该。

但实现这一点，比听起来复杂得多。这就是 OpenClaw 的第二个工程化突破：跨平台的会话统一。

问题：多渠道的身份识别

在云端 AI 服务中，会话管理相对简单：你用同一个账号登录，所有对话都绑定到这个账号。但在 OpenClaw 这样的多渠道系统中，情况完全不同。

考虑这些场景：
• 你有 3 个 WhatsApp 账号（个人、工作、家庭）
• 你在 2 台手机上登录了 Telegram
• 你在 Slack 上有多个工作空间
• 你的 macOS App 也在和 AI 对话

OpenClaw 需要回答一个核心问题：什么时候应该合并会话，什么时候应该隔离？

OpenClaw 的会话策略

OpenClaw 采用了一个巧妙的规则：

直接消息（DM）折叠到主会话
群组和频道获得独立会话

具体来说：
• DM（一对一聊天）：无论你从哪个平台发消息，只要是私聊，都会被路由到同一个"主会话"
• 群组（Group/Guild）：每个群组有独立的会话 ID，不会和 DM 混淆
• 频道（Channel）：类似群组，独立隔离

这意味着：你可以在 WhatsApp 上问"今天天气怎么样"，然后在 Telegram 上说"帮我写个邮件回复"，AI 会知道这是同一个对话。但如果你在一个 Slack 团队频道里提问，那是另一个独立的上下文。

为什么这样设计？

这是基于用户实际使用习惯的工程权衡：
• 用户期望和"自己的 AI"有一个连续的对话（无论用什么设备）
• 但在团队频道或群组中，上下文应该是隔离的（不希望工作群的讨论泄漏到私聊中）

这不是技术限制，而是用户体验设计。

上下文管理的三大挑战

会话管理的核心难题，是如何处理不断增长的上下文。AI 模型有一个"上下文窗口"限制（比如 200K tokens），而一个长期会话可能积累数十万甚至数百万 tokens 的历史。

OpenClaw 通过三种策略解决这个问题：

1. 修剪（Pruning）

在每次调用 AI 之前，OpenClaw 会自动修剪旧的工具调用结果。

比如，你让 AI 搜索了 100 个文件，生成了大量日志。这些日志对当前对话可能不再重要，可以被移除。但工具调用本身（你问了什么）会保留，AI 知道"我之前做过搜索"，只是不记得详细输出。

这是一种有损压缩，但符合人类记忆模式：我们记得"做了什么"，不一定记得"每个细节"。

2. 压缩（Compaction）

当上下文仍然超过窗口限制时，OpenClaw 会调用 AI 来总结旧对话。

流程是这样的：
• 提取最旧的一批消息（比如前 50 轮对话）
• 让 AI 写一个简短的摘要（"用户在讨论项目 X 的架构，询问了 Y 和 Z"）
• 用摘要替换原始消息
• 释放出的空间可以容纳新对话

这就像人类的长期记忆巩固：细节模糊了，但大方向还记得。

3. 轮转（Rotation）

当会话文件变得过大（比如超过 10MB），OpenClaw 会轮转文件：
• 关闭当前的 session-123.jsonl
• 创建新的 session-123.1.jsonl
• 旧文件可以归档或压缩

这是日志系统的经典模式，确保单个文件不会无限增长。

为什么用 JSONL 而非数据库？

你可能会好奇：为什么 OpenClaw 用纯文本的 JSONL 文件存储会话，而不是 SQLite 或 PostgreSQL？

这又是一个工程权衡：

JSONL 的优势：
• 可追加（Appendable）：每条新消息只需追加一行，无需重写整个文件
• 可读（Human-readable）：用文本编辑器就能查看和调试
• 工具友好（Toolable）：用 grep、jq、awk 等标准 Unix 工具处理
• 备份简单：直接复制文件即可
• 无依赖：不需要运行数据库服务

数据库的劣势（在这个场景下）：
• 查询能力对会话管理帮助不大（很少需要"查找 30 天前提到 X 的所有对话"）
• 增加了系统复杂度（需要 schema 迁移、备份策略）
• 不如文本文件直观

当然，JSONL 也有限制（比如无法高效索引），但对于 OpenClaw 的使用场景，简单性胜过灵活性。

会话的 Workspace 契约

每个 Agent 都有一个工作空间目录（Workspace），这是它的"个人文件夹"。OpenClaw 在会话开始时，会向工作空间注入一些特殊文件：

• AGENTS.md：操作指南和记忆
• SOUL.md：AI 的人格、边界、语气
• TOOLS.md：用户维护的工具使用笔记
• BOOTSTRAP.md：首次运行的仪式
• IDENTITY.md：Agent 的名字和氛围
• USER.md：用户的个人资料

这些文件是 AI 的"参考手册"。每次对话时，AI 可以读取这些文件，了解自己是谁、用户是谁、有什么能力。

这种设计的好处是：配置即代码。你可以直接编辑 SOUL.md 来改变 AI 的性格，而不需要修改程序代码。

公寓楼里的独立住户

想象一栋公寓楼：每户都有独立的门牌号、钥匙、家具，但共享同一个供水供电系统。OpenClaw 的多智能体路由就是这样的设计。

到目前为止，我们讨论的都是"单个 AI 助手"的场景。但 OpenClaw 有一个强大的能力：同一个 Gateway 可以托管多个完全隔离的 Agent。

这意味着什么？

• 你可以有一个"工作 AI"，专门处理技术问题
• 另一个"生活 AI"，负责家务安排和购物清单
• 再加一个"创意 AI"，帮你写故事和设计
• 它们都运行在同一个 OpenClaw 进程中，但彼此完全独立

每个 Agent 有自己的：
• 工作空间目录（Workspace）
• 配置文件（SOUL.md、TOOLS.md 等）
• 会话历史
• 授权配置文件
• 模型选择

这就像在同一台服务器上运行多个虚拟机，但更轻量、更高效。

为什么需要多 Agent？

你可能会想：我为什么不直接运行多个 OpenClaw 进程？

原因有三：

1. 资源效率
每个 OpenClaw 进程都要加载 Channel Providers、初始化 WebSocket 服务器、连接到 20+ 个消息平台。如果你运行 3 个进程，资源消耗是 3 倍。

多 Agent 架构只有一个 Gateway，所有 Agent 共享同一套 Channel Providers。WhatsApp 只连接一次，Telegram 只登录一次，但消息可以被路由到不同的 Agent。

2. 统一配置
单一配置文件管理所有 Agent，而不是分散在多个进程中。你可以在 ~/.openclaw/config.yaml 中定义所有 Agent 的行为。

3. 动态路由
通过绑定规则（Bindings），你可以让不同的消息源自动路由到不同的 Agent。比如：
• 工作 Slack 的消息 → 工作 Agent
• 个人 WhatsApp 的消息 → 生活 Agent
• Discord 游戏群的消息 → 创意 Agent

这是智能多租户系统（Multi-tenancy）的设计，但不是云服务的那种多租户——这是在你自己的设备上，为你自己的多个"人格"服务。

路由的优先级系统

当一条消息到达 Gateway 时，它需要决定：这条消息应该发给哪个 Agent？

OpenClaw 使用一个五级优先级系统：

优先级 1：Peer（对等绑定）
最高优先级。用于一对一的直接绑定。比如，你可以配置："来自 @alice 的 Telegram 消息，永远发给工作 Agent"。

优先级 2：Guild（群组绑定）
针对特定群组或服务器。比如，"Discord 的 #tech-support 频道，发给技术 Agent"。

优先级 3：Account（账号绑定）
针对某个消息平台的整个账号。比如，"工作 Slack 的所有消息，发给工作 Agent"。

优先级 4：Channel（渠道绑定）
针对某种消息平台。比如，"所有 WhatsApp 消息，发给生活 Agent"。

优先级 5：Default（默认 Agent）
如果上述规则都不匹配，使用默认 Agent。

这个优先级顺序是精心设计的：从最具体到最通用。先检查是否有专门的一对一绑定，如果没有，再检查群组，依此类推。

Chapter 06

安全模型：强大但不失控
给孩子自由与保护的平衡术

想象你在教孩子骑自行车：太多保护会限制他们学习，太少保护会带来危险。OpenClaw 的安全设计就是这样的平衡。

本地 AI 助手的一个核心挑战是安全性。

与云端 AI（运行在服务提供商的安全沙盒中）不同，OpenClaw 运行在你的本地设备上，能直接访问你的文件系统、运行命令、调用系统工具。这既是它的优势（能力强大），也是它的风险（潜在破坏性）。

OpenClaw 的设计哲学是："安全但不牺牲能力"。

"Security in OpenClaw is a deliberate tradeoff: strong defaults without killing capability. The goal is to stay powerful for real work while making risky paths explicit and operator-controlled."
— 摘自 VISION.md

这句话的核心意思是：默认安全，但把选择权留给用户。

为什么"默认安全"很难？

大多数软件的安全模型要么是"完全开放"，要么是"完全封闭"。但 AI 助手的场景更复杂。

完全开放的问题：
• 任何人都能通过消息平台访问你的 AI
• AI 可以执行任何命令，包括删除文件、修改系统配置
• 恶意用户可以利用 AI 作为"跳板"攻击你的设备

完全封闭的问题：
• AI 只能在沙盒里运行，无法访问真实文件或工具
• 失去了"本地助手"的核心价值——帮你完成真实任务
• 用户体验退化，AI 变成"只能聊天"的玩具

OpenClaw 需要找到第三条路：在保持能力的同时，让风险路径显式化。

安全模型的三种策略对比
策略 优点 缺点 适用场景
完全开放 无限制能力 高风险，易被滥用 仅限单用户信任环境
完全封闭 绝对安全 能力受限，无法完成真实任务 云端 AI，不信任执行环境
OpenClaw 方案 默认安全 + 可选开放 需要用户理解配置 本地 AI，用户拥有控制权

OpenClaw 的四层安全机制

第一层：访问控制（Access Control）

默认情况下，OpenClaw 不是开放的。它使用一个叫做"Pairing"的机制来控制谁能访问 AI。

DM Pairing（直接消息配对）：
• 当有人第一次通过 DM 给你的 AI 发消息时，OpenClaw 不会自动回复
• 它会提示你："用户 @alice 尝试连接，是否允许？"
• 你需要显式批准（通过 CLI 或 macOS App）
• 批准后，@alice 的用户 ID 被加入到 allowlist，后续可以自由对话

这就像你的手机：第一次连接蓝牙设备时，需要配对确认。

群组/频道的访问控制：
• 群组和频道可以单独配置 allowlist
• 你可以设置："只有管理员可以在这个 Slack 频道里 @ AI"
• 或者："这个 Discord 服务器对所有成员开放"

这种分层控制，让你可以根据不同场景调整策略。

第二层：沙盒隔离（Sandboxing）

OpenClaw 支持可选的 Docker 沙盒。

如果你启用沙盒模式：
• 每个会话运行在独立的 Docker 容器中
• AI 无法访问你的真实文件系统（除非你明确挂载目录）
• 容器结束时，所有改动被丢弃

但注意：沙盒不是强制的。因为沙盒会限制能力（比如无法直接编辑你桌面上的文件），OpenClaw 让用户选择是否启用。

这是一个经典的权衡：
• 启用沙盒：更安全，但 AI 的实用性降低
• 不启用沙盒：AI 能完成真实任务，但需要你信任它

第三层：Owner-only 命令（特权操作）

某些敏感操作被标记为"Owner-only"：
• 修改配置文件
• 添加/删除 Agent
• 修改 allowlist
• 关闭 Gateway

只有"Owner"（在配置中定义的主用户）可以执行这些命令。即使其他用户被配对并允许访问 AI，他们也无法执行这些特权操作。

这类似于 Unix 的 sudo 机制：日常操作无需特权，危险操作需要提权。

第四层：Elevated Mode（提权模式）

OpenClaw 还有一个"Elevated Mode"（提权模式），允许 AI 执行更危险的操作，比如：
• 安装软件包
• 修改系统配置
• 访问网络敏感 API

进入 Elevated Mode 需要：
• 用户显式授权（通过命令或配置）
• 可选的二次确认（比如输入密码）

这确保了即使 AI 被"社会工程攻击"（用户被诱导让 AI 做危险操作），也有一道额外的防护。

安全配置对比：开放 vs 严格
过于开放的配置（高风险）
security:
  pairing_required: false  # 任何人都能访问
  sandbox: false           # 无沙盒保护
  owner_only_cmds: []      # 无特权命令限制
  elevated_mode: always    # 永久提权

极度危险：任何人都能让 AI 执行任意命令，包括删除文件、修改系统。

OpenClaw 推荐配置
security:
  pairing_required: true   # 需要配对
  sandbox: optional        # 用户可选
  owner_only_cmds:         # 特权命令受限
    - config:update
    - agent:create
  elevated_mode: manual    # 需要显式授权

平衡：默认安全，但保留能力。用户可以根据需求调整策略。

为什么不强制沙盒？

很多人会问：既然沙盒更安全，为什么不强制所有 Agent 都运行在沙盒里？

答案是：因为沙盒会杀死实用性。

想象这些场景：
• 你让 AI "帮我修改桌面上的 report.docx，添加一个结论段落"
• 你让 AI "运行我的开发项目，看看测试是否通过"
• 你让 AI "帮我整理下载文件夹，把所有 PDF 移动到 Documents"

如果 AI 运行在沙盒里，它看不到你的桌面、开发项目、下载文件夹。你需要手动挂载这些目录到容器中，但这样一来：
• 配置变得复杂（普通用户很难理解 Docker volume 挂载）
• 体验变差（"为什么 AI 找不到我的文件？"）
• 失去了"无缝助手"的感觉

OpenClaw 的选择是：默认不启用沙盒，但提供工具让你轻松启用。

如果你需要高安全性（比如让陌生人使用你的 AI，或者处理不信任的代码），可以开启沙盒。但对于日常个人使用，沙盒往往弊大于利。

实际场景的权衡

让我们看几个真实场景，以及如何配置安全策略。

场景 1：个人使用，完全信任

配置：
• Pairing: 关闭（只有你一个人用，无需配对）
• Sandbox: 关闭（需要 AI 访问真实文件系统）
• Owner-only: 启用（防止误操作）
• Elevated Mode: 手动（敏感操作需要确认）

风险：低。只要你不被社会工程攻击，系统是安全的。

场景 2：团队共享，多人访问

配置：
• Pairing: 启用（每个团队成员需要配对）
• Sandbox: 可选（看团队信任程度）
• Owner-only: 严格（只有管理员能修改配置）
• Elevated Mode: 完全关闭（不允许任何人提权）

风险：中。需要确保团队成员不会恶意利用 AI。

场景 3：公开演示，不信任环境

配置：
• Pairing: 启用
• Sandbox: 强制（每个会话独立容器）
• Owner-only: 严格
• Elevated Mode: 完全禁用
• 只读文件系统（AI 无法修改任何文件）

风险：最低。但能力也最受限。

为什么这是工程化突破？

OpenClaw 的安全模型的创新，不在于发明了新技术，而在于找到了正确的权衡点。

大多数本地 AI 项目的困境：
• 要么完全忽略安全（"反正只有我自己用"）
• 要么过度限制（"为了安全，只能在沙盒里运行"）

OpenClaw 的创新：
• 分层的安全模型，用户可以根据场景调整
• 默认安全但不牺牲能力
• 风险路径显式化（需要你主动关闭保护，而不是默认开放）
• 配置驱动，无需修改代码

这种设计让 OpenClaw 既适合个人信任环境（充分发挥能力），也适合多人共享环境（保持安全边界）。

语音与画布：多模态交互的魔法
超越文字的 AI 体验

如果 AI 只能打字聊天，那就像只能写信的朋友——有用，但不够自然。OpenClaw 带来了语音和可视化能力。

到目前为止，我们讨论的都是 OpenClaw 的"引擎"部分——架构、会话、路由、安全。但 AI 助手的价值，最终体现在用户体验上。

OpenClaw 不仅仅是"文字聊天机器人"。它从设计之初就支持多模态交互：语音、可视化、图像、视频。这一章我们快速浏览这些能力。

Voice Wake：唤醒你的 AI

在 macOS 和 iOS 上，OpenClaw 支持唤醒词检测（Voice Wake）。

这意味着你可以：
• 说"Hey Molty"（或你自定义的唤醒词）
• AI 被激活，开始监听
• 你说出你的问题或请求
• AI 用语音回复

这种交互模式让 AI 感觉更像"一直在你身边的助手"，而不是"需要打开 App 才能用的工具"。

技术实现：OpenClaw 使用一个名为 Swabble 的独立守护进程，用 Swift 编写，基于 macOS 的 Speech.framework。它在后台持续监听，当检测到唤醒词时，唤醒 Gateway 并开始会话。

Talk Mode：连续对话

除了唤醒词，OpenClaw 还支持Talk Mode——连续的语音对话模式。

在这个模式下：
• 你不需要每次都说唤醒词
• AI 会持续监听你的语音
• 自动识别你何时说完（通过语音停顿检测）
• 立即回复，形成流畅的对话

这就像和真人对话一样自然。

Live Canvas：可视化工作空间

文字和语音都有局限性。对于某些任务，可视化才是最高效的交互方式。

OpenClaw 的 Live Canvas 是一个由 AI 驱动的动态 UI 空间。AI 可以：
• 推送可视化内容（图表、表格、代码编辑器）
• 创建交互式控件（按钮、滑块、输入框）
• 实时更新内容（比如显示任务进度）

想象这些场景：
• 你让 AI "帮我分析这份销售数据"，AI 在 Canvas 上绘制了柱状图和趋势线
• 你让 AI "生成一个待办清单"，AI 在 Canvas 上创建了可勾选的列表
• 你让 AI "帮我调试这段代码"，AI 在 Canvas 上高亮了错误行，并提供了交互式修复建议

Canvas 不是"静态的输出区域"，而是 AI 的第二个表达通道。

A2UI：AI-to-UI 协议

Canvas 的底层是一个叫做 A2UI（AI-to-UI）的协议。

它定义了三种基本操作：
• Push：AI 向 Canvas 推送新内容
• Reset：清空 Canvas，重新开始
• Eval：执行 JavaScript 代码，动态更新 UI

这个协议让 AI 能够"控制"用户界面，而不仅仅是"输出文本"。

媒体管道：图像、视频、音频

OpenClaw 内置了一个媒体管道（Media Pipeline），可以处理各种格式的媒体文件：

• 图像：JPEG、PNG、WebP、GIF
• 视频：MP4、MOV、WebM
• 音频：MP3、WAV、AAC

用户可以直接在 WhatsApp、Telegram 等平台发送图片或视频，AI 会自动处理并理解内容。比如：
• 发一张菜单照片，问"推荐点什么？"
• 发一段视频，问"这段视频的主要内容是什么？"
• 发一张手绘草图，让 AI "把这个想法变成代码"

Chapter 08

扩展生态：从核心到无限可能
为什么可扩展性是工程化的终极体现

一个好的系统，不是"功能齐全"，而是"容易添加新功能"。OpenClaw 的插件化架构，让它像乐高积木一样可组合。

到目前为止，我们讨论的都是 OpenClaw 的"核心能力"：Gateway、会话管理、路由、安全。但让 OpenClaw 真正强大的，是它的可扩展性。

这是本指南的最后一个重点章节，我们会聚焦一个工程化问题：如何设计一个系统，让它既有清晰的核心，又能无限扩展？

为什么选择 TypeScript？

在第二章我们提到过，OpenClaw 选择 TypeScript 是一个故意的工程决策。现在我们深入探讨这个选择的价值。

创始人 Peter Steinberger 在 VISION.md 中写道：

"OpenClaw is primarily an orchestration system: prompts, tools, protocols, and integrations. TypeScript was chosen to keep OpenClaw hackable by default. It is widely known, fast to iterate in, and easy to read, modify, and extend."

这段话的关键词是：hackable by default（默认可黑客化）。

对比其他选择：
• Python：虽然也易读，但缺少类型安全，大型项目容易出错
• Rust：性能极佳，但学习曲线陡峭，普通开发者难以修改
• Go：简洁高效，但语言表达力有限，不适合复杂的编排逻辑

TypeScript 的优势：
• 广泛使用：几乎所有前端开发者都会 TypeScript，社区庞大
• 快速迭代：修改代码、运行、测试的循环极快
• 易于阅读：非专业开发者也能看懂大部分代码
• 类型安全：TypeScript 的类型系统能在编译时捕获错误
• 生态丰富：npm 上有海量的库可以直接使用

这个选择的核心逻辑是：对于编排系统，可黑客性比性能更重要。

OpenClaw 的瓶颈不在 TypeScript 的执行速度（大部分时间都在等待 AI 模型返回），而在用户能否轻松地修改和扩展它。

插件化架构：核心保持精简

OpenClaw 的设计哲学是：核心做最少的事，能力通过插件扩展。

核心（Core）只包含：
• Gateway（路由和控制平面）
• Agent Runtime（基于 pi-mono 的执行引擎）
• 基础 Channel Providers（WhatsApp、Telegram、Slack）
• 配置系统和事件总线

所有其他能力都是扩展：
• 40+ 额外的 Channel Providers（Discord、Signal、Matrix...）→ extensions/
• 54+ Skills（GitHub、Spotify、浏览器控制...）→ skills/
• 自定义工具和插件 → Plugin SDK

这种设计的好处是：
1. 核心保持简单：不会因为支持太多平台而变得臃肿
2. 按需加载：只加载你需要的扩展，节省资源
3. 社区贡献：任何人都可以创建新的 Channel Provider 或 Skill
4. 独立更新：扩展可以独立于核心更新，不需要重新编译整个系统

Plugin SDK：标准化的扩展接口

OpenClaw 提供了一个 Plugin SDK，让开发者可以创建自己的插件。

插件可以做什么？
• 添加新的 Channel Provider（比如支持微信、钉钉）
• 添加新的工具（比如控制智能家居、访问特定 API）
• 修改 Gateway 的行为（比如自定义路由规则）
• 注入新的上下文管理策略

Plugin SDK 的设计原则：
1. 最小接口：插件只需要实现几个核心方法，不需要理解整个系统
2. 类型安全：TypeScript 类型定义确保插件和核心兼容
3. 热加载：插件可以在运行时加载和卸载，无需重启 Gateway
4. 沙盒隔离：插件运行在独立的上下文中，防止相互干扰

Skills 系统：预定义的能力包

如果说插件是"底层扩展机制"，那么 Skills 就是"高层能力包"。

Skill 是一个预定义的功能模块，用户可以通过简单的命令安装和启用。OpenClaw 内置了 54+ 个 Skills，比如：

• github：查询 GitHub 仓库、创建 Issue、审查 PR
• spotify-player：控制 Spotify 播放、搜索音乐
• slack：发送 Slack 消息、管理频道
• browser：控制浏览器、抓取网页内容
• cron：定时任务调度
• node-host：控制其他设备上的 OpenClaw 节点

Skills 的特点：
• 声明式配置：通过 YAML 文件定义 Skill 的行为
• 依赖管理：Skills 可以依赖其他 Skills
• 版本控制：Skills 有版本号，可以更新和回滚
• 权限控制：Skills 需要声明它们需要的权限（比如文件系统访问）

ClawHub：社区技能注册表

OpenClaw 还计划推出 ClawHub，一个社区驱动的 Skills 注册表。

类似于 npm、PyPI、crates.io，ClawHub 让用户可以：
• 搜索和发现社区创建的 Skills
• 一键安装 Skills
• 发布自己的 Skills 供他人使用
• 评价和评论 Skills

这将形成一个正向循环：
1. 开发者创建 Skills 解决自己的问题
2. 发布到 ClawHub 让他人受益
3. 更多人使用 → 更多反馈 → Skill 变得更好
4. 生态繁荣 → 吸引更多开发者

为什么这是工程化突破？

可扩展性看起来是"软件工程 101"，但真正做好并不容易。

大多数 AI 助手项目的问题：
• 要么是"单体应用"，所有功能都硬编码在核心中
• 要么是"过度抽象"，扩展接口如此复杂，以至于没人愿意写插件

OpenClaw 的平衡点：
• 核心足够精简，不会因为支持太多平台而臃肿
• 扩展接口足够简单，普通开发者也能写插件
• 三层扩展方式（内置 Skills、自定义 Skill、Plugin），适应不同需求
• TypeScript 的选择，让代码易读易改

这种设计让 OpenClaw 不仅仅是"一个 AI 助手"，而是一个可编程的 AI 基础设施。

问题：如何设计一个既有清晰核心，又能无限扩展的系统？

方案：核心精简（Gateway + Agent）+ 插件化架构（Extensions + Skills）+ 社区生态（ClawHub）+ TypeScript 的可黑客性。

价值：核心保持简单 + 按需加载扩展 + 三层扩展方式适应不同需求 + 社区驱动的持续创新。

类比：乐高积木——基础块有限，但组合方式无限；每个人都能创造自己的作品。

本章要点

1. TypeScript 的选择：可黑客性 > 性能,适合编排系统

3. 四层结构：核心 → 扩展 → Skills → 社区
