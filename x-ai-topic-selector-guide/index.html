开篇

为什么需要这个工具?

想象你是一位内容创作者,每天早上打开 Twitter,时间线上滚动着数千条推文。有人分享最新的 AI 模型突破,有人讨论行业格局变化,有人发布开源工具更新。你知道其中一定有值得深挖的选题,但问题是——哪些真正值得你花时间?

这不是时间管理问题,而是信息筛选能力的瓶颈。人类一天能消化的信息量是有限的,但社交媒体产生内容的速度是指数级的。传统做法是靠"感觉"和"经验"快速扫一遍,点赞收藏几条,然后……这些收藏的推文往往就躺在那里再也没打开过。

X-AI-Topic-Selector 解决的正是这个痛点。它不是另一个 RSS 阅读器,也不是简单的关键词过滤工具,而是一个智能选题助手——它会像你的研究实习生一样,帮你完成最耗时的初筛工作,然后用结构化报告呈现最有价值的内容。

这个工具由微信公众号"懂点儿AI"开发,作者本身就是内容创作者,深知选题筛选的痛苦。所以它的设计哲学不是"自动化一切",而是"把人类最不擅长的重复劳动交给机器,把需要判断力的决策留给人"。

Chapter 01

信息洪流中的淘金者

在社交媒体时代,信息不是稀缺资源,注意力才是。

假设你关注了 500 个 Twitter 账号,每个账号平均每天发 3 条推文,那就是 1500 条内容。即使你用 2 倍速刷,每条只看 5 秒,也需要 2 个多小时才能刷完——而这还没算上那些多推文线程(Twitter Thread)和需要点开链接才能看完整内容的推文。

更糟糕的是,数据指标会骗人。一条推文有 10 万点赞,可能只是因为它迎合了某种情绪,而不是真的有深度。一个开源项目刚发布时互动数不高,但可能是你所在领域的技术突破。传统的"按热度排序"根本无法解决这个问题。

内容创作者的三重困境

对于需要持续输出内容的创作者来说,选题筛选面临三个核心挑战:

1. 时效性陷阱: 热点来得快去得也快。你看到一条新闻,觉得有价值,收藏了准备晚点写。等到晚上有时间了,发现已经有 20 篇解读文章,你的角度已经不新鲜了。

2. 信息孤岛: 有价值的内容分散在不同平台、不同账号、不同话题下。你可能会错过那些藏在 List 里的小众但优质的内容,因为它们从未出现在你的主时间线。

3. 决策疲劳: 当你需要在几百条推文中快速判断"这个值得写吗?",大脑会进入一种类似"选择困难症"的状态。到最后,你可能因为太累而放弃筛选,或者随便选一个看起来还行的话题。

这些数字背后是一个残酷的现实:你花了 3 个小时筛选,最终只用上了不到 1% 的内容。剩下 99% 的时间都消耗在了"扫一眼,判断不值得,下一条"的重复劳动上。

X-AI-Topic-Selector 的价值就在这里。它不是要替代你的判断力,而是要把这 3 个小时压缩到 10 分钟。它帮你完成"扫一眼,判断不值得"这个动作,然后把真正需要你判断力的 Top 10 候选选题,用结构化报告的形式呈现给你。

核心洞察

1. 信息过载的本质是注意力稀缺,不是内容稀缺
2. 数据指标(点赞/转发)无法反映内容对特定领域的深度价值
3. 人工筛选的 99% 时间浪费在重复性的"判断不值得"动作上
4. 工具的目标不是替代判断力,而是放大判断力的效率

Chapter 02

双模式智慧:扫描与摘要的分工哲学

同样是筛选推文,为什么需要两种完全不同的工作方式?

想象你在逛书店。有两种场景:

场景 A: 你走进一家大书店,面对几千本书,你需要快速扫过每个书架,根据封面、标题、简介快速筛选出 10 本可能感兴趣的书,然后带回家慢慢读。

场景 B: 你已经在过去一个月里收藏了 50 本书的书签,现在你需要坐下来,把每本书的核心观点提炼出来,做成读书笔记,方便日后查阅。

这两种场景的目标、流程、时间投入完全不同。X-AI-Topic-Selector 的双模式设计正是源于这个洞察:从海量内容中快速初筛,和对已筛选内容进行深度分析,是两种不同的工作,不应该用同一套流程。

Scan Mode:快速淘金模式

扫描模式(Scan Mode)适用于从未知领域快速发现机会。你可能在逛某个 Twitter List,或者浏览主时间线,不确定今天会有什么有价值的内容。这种场景下,你需要的是:

  • 快速抓取: 自动滚动加载,抓取最多 200 条推文(可配置)
  • 粗筛过滤: 用关键词、分类、基础数据指标(点赞/转发数)快速排除明显不相关的内容
  • AI 快速评分: 用 AI 给剩余内容打分,提取 Top N(默认 10 条)
  • 效率优先: 不需要完整阅读每条推文,只看标题和摘要就够了

这个模式的核心思想是"漏斗式筛选":从 1500 条推文筛到 200 条(抓取限制),再筛到 50 条(关键词过滤),再筛到 10 条(AI 评分 Top N)。每一层都在快速排除噪音,保留信号。

Bookmark Mode:深度摘要模式

书签摘要模式(Bookmark Mode)则完全不同。它假设你已经完成了初筛——那些躺在你 Twitter Bookmarks 里的推文,都是你当时觉得"可能有价值"而收藏的。现在的问题不是"哪些值得看",而是"这些推文到底在讲什么?我为什么要收藏它?"

这个模式的工作方式:

  • 全量处理: 提取所有书签(默认最多 50 条,可配置更多),不做 Top N 截断
  • 深度分析: 强制使用 AI 分析每一条推文,生成中文标题、摘要、关键词标签
  • 结构化输出: 按日期分组,生成类似"每日精选"的报告,方便回溯
  • 内容完整: 如果推文被截断,会自动导航到原推文页面获取完整文本

注意这里的区别:Scan Mode 是"从海量内容中找 10 个宝藏",Bookmark Mode 是"把 50 个可能的宝藏都仔细鉴定一遍"。前者追求效率,后者追求完整性。

为什么不能用一套流程?

你可能会想:为什么不能让 Scan Mode 也做深度分析?或者让 Bookmark Mode 也做 Top N 筛选?答案是成本与收益的权衡。

如果你对 200 条推文都做深度 AI 分析,以每条推文消耗 500 tokens 计算(包括输入和输出),那就是 10 万 tokens。按 Gemini 2.0 Flash 的价格($0.075/1M tokens 输入,$0.30/1M tokens 输出),一次扫描的成本就是几美分——听起来不多,但如果你每天扫描 3 次,一个月就是几美元。更重要的是,你根本不需要知道第 150 名推文的详细分析,你只关心 Top 10。

相反,Bookmark Mode 处理的是"你已经收藏的"内容,数量通常在 20-50 条之间。这些内容都值得深度分析,因为它们已经通过了你的初筛。在这种场景下,省略 AI 分析反而是浪费——你收藏了却不知道它讲什么,那收藏的意义何在?

设计哲学

1. 快速初筛(Scan)和深度分析(Bookmark)是两种不同的认知过程
2. Scan Mode 用"漏斗式筛选"在效率和成本间取得平衡
3. Bookmark Mode 强制深度分析,因为已收藏内容值得完整理解
4. 工具设计应该匹配用户的实际工作流,而不是追求"功能大而全"

Chapter 03

无依赖的 Chrome 自动化

为什么不用 Puppeteer?当你直接和浏览器对话,世界会变得更简单。

大多数开发者在需要浏览器自动化时,第一反应是 npm install puppeteer。这很合理——Puppeteer 是 Google 官方维护的工具,API 友好,文档齐全。但 X-AI-Topic-Selector 的作者选择了一条不同的路:直接使用 Chrome DevTools Protocol (CDP),零依赖控制浏览器。

这个决策背后有三个关键考量。

考量一:启动速度与资源占用

Puppeteer 本质上是 CDP 的封装层,它提供了更友好的 API,但也引入了额外的抽象开销。更重要的是,Puppeteer 默认会下载一个完整的 Chromium 浏览器(约 170MB),并且每次启动时都会创建一个全新的浏览器实例。

X-AI-Topic-Selector 的做法是使用你系统中已有的 Chrome 浏览器。它会自动查找 Chrome 的安装路径(支持 macOS、Linux、Windows),然后用 --remote-debugging-port 参数启动,直接通过 WebSocket 连接到 CDP。

这带来两个好处:不需要额外下载浏览器,以及可以复用已登录的用户会话。后者尤其关键——Twitter 需要登录才能访问完整功能,如果每次都启动全新浏览器,你就得处理登录、验证码、会话保持等一堆麻烦事。

考量二:依赖链的脆弱性

当你引入 Puppeteer,你实际上引入了一个复杂的依赖树。如果你用的是 Bun 运行时(X-AI-Topic-Selector 的选择),Puppeteer 的某些 Node.js 特定依赖可能会有兼容性问题。而 CDP 是一个标准化的 JSON-RPC 协议,只要浏览器支持,任何语言、任何运行时都能用。

这个项目的依赖列表非常克制:只有 2 个运行时依赖(@google/generative-ai 和 openai,用于 AI 评分)。所有浏览器自动化代码都是手写的 CDP 调用,不超过 300 行。

考量三:对登录状态的需求

Twitter 的很多功能需要登录才能访问,比如查看完整的推文线程、访问 Bookmarks、读取 Lists。如果用 Puppeteer 的默认配置,每次启动都是一个"无痕浏览"的全新会话,你得在代码里处理登录逻辑——输入用户名、密码、可能的两步验证……这不仅麻烦,还涉及安全风险(你总不想把 Twitter 密码硬编码在脚本里)。

X-AI-Topic-Selector 的方案是使用 --user-data-dir 参数,指向一个持久化的用户配置目录。第一次运行时,浏览器会打开让你手动登录 Twitter。登录成功后,会话状态(cookies、localStorage)会保存到这个目录。之后每次启动,都会自动恢复登录状态,就像你平时打开 Chrome 一样。

核心实现:Chrome 启动与 CDP 连接
// 1. 启动 Chrome 进程
const chrome = spawn(chromePath, [
  `--remote-debugging-port=${port}`,      1
  `--user-data-dir=${profileDir}`,        2
  '--no-first-run',
  '--no-default-browser-check'
]);

// 2. 等待 CDP 端点就绪
const wsUrl = await waitForDevToolsUrl(port);

// 3. 建立 WebSocket 连接
const cdp = new CdpConnection(wsUrl);      3

// 4. 通过 CDP 执行 JavaScript
const result = await cdp.send('Runtime.evaluate', {
  expression: `
    Array.from(document.querySelectorAll('[data-testid="tweet"]'))
      .map(el => ({ text: el.innerText, ... }))
  `
});                                        4

2 指定用户配置目录,保存登录状态和浏览器设置

三层抓取策略:应对 Twitter 的复杂场景

光有 CDP 连接还不够,Twitter 的页面结构给自动化带来了三个挑战:

挑战 1:无限滚动加载。Twitter 使用虚拟滚动,一次只渲染可视区域的推文。你需要不断向下滚动,等待新内容加载,直到达到目标数量或没有新推文出现。X-AI-Topic-Selector 的做法是循环执行 window.scrollTo(0, document.body.scrollHeight),每次滚动后等待 2 秒,如果连续 3 次滚动后推文数量没变化,就认为已经到底了。

挑战 2:多推文线程(Thread)。很多有价值的内容是以"线程"形式发布的——一个主推文下面跟着 5-10 条回复,共同构成完整观点。如果只抓取主推文,你会丢失大量上下文。工具的策略是:识别出所有包含"显示更多回复"的推文,并发(最多 3 个)点击展开,等待内容加载后再提取。

挑战 3:文本截断。Twitter 在时间线上只显示推文的前 280 个字符,后面是"显示更多"链接。对于长推文,如果不展开,你拿到的就是残缺的文本。X-AI-Topic-Selector 的处理方式是:先抓取所有推文的基本信息,然后对需要完整文本的推文,单独导航到它的详情页,提取完整内容。

工程权衡

1. 直接使用 CDP 牺牲了 API 友好度,换来了零依赖和更快的启动速度
2. 使用系统 Chrome + 持久化配置目录,避免了复杂的登录流程
3. 三层抓取策略确保内容完整性,但增加了抓取时间(线程展开和文本补全都需要额外的页面导航)
4. 这套方案适合"间歇性运行"的 CLI 工具,不适合需要 24/7 运行的爬虫服务

Chapter 04

评分系统:数据与 AI 的双轨制

点赞数不代表价值,但 AI 也不是万能的——最优解是让它们互相补充。

假设有两条推文:

推文 A: 一张猫咪图片配文"周一早上的心情",2 万点赞,5000 转发,3000 评论。
推文 B: 一篇关于新发布的开源 AI 框架的深度分析,500 点赞,50 转发,20 评论。

如果只看数据指标,推文 A 完胜。但如果你是 AI 领域的内容创作者,推文 B 才是你需要的"高价值选题"。这就是为什么 X-AI-Topic-Selector 需要双轨评分系统:数据评分负责量化"社交影响力",AI 评分负责判断"内容深度"。

数据评分:快速筛选的第一道关卡

数据评分公式非常直接:

评分 = 点赞数 × 1 + 转发数 × 3 + 评论数 × 2 + 浏览量 × 0.01

这个公式的权重设计有讲究。转发权重最高(×3),因为转发意味着用户愿意把内容分享到自己的时间线,这比单纯点赞的认可度更高。评论权重居中(×2),因为评论表示内容引发了讨论,但也可能是争议性内容(高评论不一定是好事)。浏览量权重极低(×0.01),因为浏览量很容易被标题党推高,只能作为辅助参考。

数据评分的核心作用是在 AI 评分之前做粗筛。当你从时间线抓取了 200 条推文,数据评分可以快速排除那些明显没人关注的内容(比如只有个位数互动的推文),把候选池缩小到 50-100 条,然后再交给 AI 做深度分析。

AI 评分:三维价值评估框架

AI 评分系统使用了一个三维评估框架,专门针对 AI 资讯类内容设计:

维度 1:创新性(Innovation) · 1-5 分
关注"这个内容有多新?"。高分标准包括:首次披露的信息、技术突破、行业格局变化、独特的视角。比如,某个 AI 公司宣布新模型发布,创新性可能是 4-5 分;而对已有模型的常规使用教程,可能只有 2-3 分。

维度 2:实用性(Practicality) · 1-5 分
关注"读者能否马上用起来?"。高分标准包括:可执行的代码示例、工具推荐、问题解决方案、工作流优化技巧。纯理论讨论的实用性较低,而"如何用 5 行代码实现 XX 功能"的实用性很高。

维度 3:影响力(Impact) · 1-5 分
关注"这个内容的潜在影响范围有多大?"。高分标准包括:行业标准变化、大公司战略调整、开源项目重大更新、监管政策出台。个人经验分享的影响力较低,而"OpenAI 宣布新定价策略"的影响力很高。

每条推文的AI 总分 = 创新性 + 实用性 + 影响力,满分 15 分。这个框架的优势在于:它不是简单判断"好"或"不好",而是从三个维度量化内容价值,让你可以根据当前需求调整权重(比如,如果你在准备技术教程,可以优先看实用性高的推文)。

批处理与成本优化

AI 评分的挑战在于成本和速度的平衡。如果对 100 条推文逐条调用 AI API,总耗时可能超过 2 分钟,且 API 调用次数过多会增加成本和被限流的风险。

X-AI-Topic-Selector 的优化策略是批处理(Batching):每次把 5 条推文打包成一个 Prompt,发送给 AI 模型,要求返回一个 JSON 数组,包含 5 条推文的评分结果。同时,最多并发 2 个批次,也就是同时处理 10 条推文。

这个设计在效率和稳定性之间取得平衡:批次太大(比如 20 条),AI 模型容易"跑偏"(后面的推文评分质量下降);批次太小(比如 1 条),API 调用次数太多。实际测试发现,5 条/批 + 并发 2 批是最优配置,可以在 30-60 秒内完成 50 条推文的 AI 评分。

Chapter 05

多 AI 提供商的统一抽象层

当 AI 成为基础设施,你需要的不是供应商锁定,而是自由切换的能力。

2024 年 12 月,X-AI-Topic-Selector 发布 v1.4.0,引入了一个看似简单但影响深远的功能:多 AI 提供商支持。用户现在可以在 Gemini、OpenAI、DeepSeek 等不同 AI 服务之间自由切换,甚至让工具自动检测并选择可用的服务。

这个功能背后是一个经典的软件工程问题:如何在保持代码简洁的前提下,支持多个接口不同的外部服务?答案是抽象层(Abstraction Layer)。

为什么需要多提供商支持?

在 v1.4.0 之前,工具只支持 Google Gemini API。这在大多数情况下没问题,但用户反馈了三个痛点:

痛点 1:地域限制。Gemini API 在某些国家和地区不可用,或者需要特殊的网络配置。OpenAI 和 DeepSeek 的可用性更广泛。

痛点 2:成本差异。不同 AI 服务的定价差异巨大。以 100 万 tokens 为例:Gemini 2.0 Flash 约 $0.075(输入),OpenAI GPT-4o Mini 约 $0.15,而 DeepSeek 只需 $0.014。对于每天运行多次的用户,这个差异很可观。

痛点 3:服务稳定性。没有任何 AI 服务能保证 100% 可用。当 Gemini API 返回 503 错误时,如果工具能自动切换到备用服务,用户体验会好得多。

抽象层的设计哲学

X-AI-Topic-Selector 的解决方案是创建一个 ai-client.ts 模块(139 行),定义了一个统一的接口:

  • 输入:一段文本(推文内容)+ 一个 Prompt(评分指令)
  • 输出:一个 JSON 对象(包含评分结果)
  • 实现:根据环境变量自动选择 Gemini、OpenAI 或兼容 OpenAI 的服务(如 DeepSeek)

调用方(评分系统)不需要知道底层用的是哪个 AI 服务,只需要调用 generateContent(prompt) 方法。这就是抽象层的力量:把"选择哪个 AI"的决策从业务逻辑中剥离出来。

自动检测逻辑

当用户设置 --ai-provider auto-detect(默认值)时,工具会按以下优先级选择 AI 服务:

  1. 检查 GEMINI_API_KEY:如果存在且非空,使用 Gemini
  2. 检查 OPENAI_API_KEY:如果存在,再检查 OPENAI_API_BASE:
    • 如果 OPENAI_API_BASE 包含"deepseek",识别为 DeepSeek
    • 否则识别为 OpenAI
  3. 都不存在:抛出错误,提示用户配置环境变量

这个逻辑的巧妙之处在于:DeepSeek 和其他兼容 OpenAI 的服务不需要特殊支持。只要它们实现了 OpenAI 的 Chat Completions API,就可以通过设置 OPENAI_API_BASE 来使用。这种设计让工具具备了"开放扩展"的能力——未来出现新的兼容服务,用户无需等待工具更新,直接配置环境变量即可。

错误处理与降级策略

抽象层还负责处理 AI 服务的异常情况。当 API 调用失败时(比如网络超时、限流、返回格式错误),工具不会直接崩溃,而是:

  • 记录错误日志:输出详细的错误信息,方便调试
  • 返回默认评分:创新性 3 分,实用性 3 分,影响力 3 分(总分 9/15)
  • 继续处理下一条:单条推文评分失败不影响整体流程

这种"优雅降级"的设计理念是:AI 评分是增强功能,不是核心依赖。即使 AI 完全不可用,工具仍然可以用数据评分模式完成筛选任务。

Chapter 06

从数据到洞察:报告生成的艺术

原始数据是石头,结构化报告是雕塑——关键在于如何雕刻。

当工具完成推文抓取、评分、排序后,最后一步是生成报告。这一步看似简单(不就是把数据输出成 Markdown 吗?),但实际上决定了用户体验的 90%。一份好的报告应该让用户一眼就能找到自己需要的信息,而不是盯着一堆 JSON 数据发呆。

X-AI-Topic-Selector 的报告生成模块(report-generator.ts,470 行)做了大量细节优化,核心理念是:把"数据呈现"升级为"信息传达"。

Scan Mode 报告:聚焦 Top N 精选

扫描模式的报告结构非常清晰,分为四个部分:

Part 1:执行摘要
顶部用表格展示关键指标:抓取推文总数、过滤后数量、AI 评分数量、最终推荐数量。这让用户快速了解"漏斗"的每一层筛选效果。

Part 2:数据可视化
使用 Mermaid 图表(饼图、柱状图)展示:分类分布、作者分布、评分分布。如果 Mermaid 渲染失败(比如在不支持的 Markdown 查看器中),会自动回退到 ASCII 纯文本图表,确保兼容性。

Part 3:Top N 推荐列表
每条推文包含:排名、标题(AI 生成的中文标题)、作者、发布时间、互动数据、评分(数据分和 AI 分)、摘要、关键词标签、原文链接。信息密度高但不杂乱,用 Markdown 的层级结构清晰呈现。

Part 4:配置记录
底部附录记录本次执行的所有参数:URL、评分模式、Top N 数量、关键词过滤、AI 提供商等。这对于复现结果或调整策略非常重要。

Bookmark Mode 报告:按日期分组的知识管理

书签摘要模式的报告设计完全不同。因为用户收藏的推文可能来自不同时期,报告会按发布日期分组,生成类似"每日精选"的结构:

  • 日期分组标题:比如"2026 年 3 月 11 日 - 5 条推文"
  • 每条推文完整分析:不再做 Top N 筛选,所有书签都包含在报告中
  • 中文标题和摘要:即使原推文是英文,也会生成中文标题和摘要,方便中文用户快速浏览
  • 关键词标签聚合:报告末尾会汇总所有标签,显示高频主题

这种设计的理念是:Bookmark 是个人知识库,报告应该像"读书笔记"而非"排行榜"。
