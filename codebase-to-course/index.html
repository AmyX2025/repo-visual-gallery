「Skill」究竟是什么

你可能想知道：Skill 到底是什么？把它想象成雇用一位专业合同工。

项目承包商的比喻

你是项目经理，Claude 是万能型全才。而 Skill 是一份详尽的岗位说明书，把 Claude 变成专科专家——「当有人让你做 X 的时候，按照这些具体步骤来执行。」SKILL.md 就是那份说明书。

当你输入「把这个做成课程」，Claude Code 检测到触发短语，加载 SKILL.md 文件——从此刻起，它会严格按照文件里的指令行事：分析代码库、设计课程大纲、读取参考文件、生成 HTML 输出。

## Who This Is For

The target learner is a "vibe coder"
— someone who builds software by
  instructing AI coding tools in
  natural language, without a
  traditional CS education.

Assume zero technical background.
Every CS concept needs to be
explained in plain language.

把这句话写进 SKILL.md，Claude 在生成任何课程时都会自动保持正确的表达方式——零假设知识，遇到术语必加解释。

03

秘方

SKILL.md 是一份详尽的工作说明书。以下是它逐步指导 Claude 完成的每一步。

四个阶段，一份精美输出

SKILL.md 将整个工作拆解为四个阶段，必须依次完成。就像烤面包——你不能在面团还没发酵的时候就开始整形。

1

代码库分析

读完所有关键文件。追踪数据流向。梳理「演员表」（各组件、模块、服务）。提取技术栈、主要用户旅程以及任何巧妙的工程设计模式。这个阶段越深入，课程质量越高。

2

课程大纲设计

在内部完成 5–8 个模块的设计——不向用户寻求审批。结构永远从用户可感知的外部行为出发（「它做什么」），逐步深入到底层代码。每个模块必须与某项实用技能挂钩。

3

构建课程

生成 HTML——但要按顺序来：先搭框架，再逐个模块填充。绝不一次性写完所有模块（质量会下降）。在写任何一个标签之前，先读完 design-system.md 和 interactive-elements.md。

4

审阅与开启

在浏览器中打开课程，带用户浏览已构建的内容，征询反馈。课程是对话的起点，不是默默交付的终点产品。

第一阶段近观：读懂代码库

第一阶段是最重要的。浅尝辄止的分析只能产出浅薄的课程。以下是 SKILL.md 指示 Claude 要提取的内容——以及每一项对学习者的实际意义。

SKILL.md — Phase 1 
### Phase 1: Codebase Analysis

What to extract:
- The main "actors" and their responsibilities
- The primary user journey
- Key APIs, data flows, communication patterns
- Clever engineering patterns
- Real bugs or gotchas
- The tech stack and why each piece

Figure out what the app does yourself
by reading the README, main entry
points, and UI code. Don't ask
the user to explain the product.
白话文解读

「演员」= 应用里实际负责「做事」的主要组件或文件，像舞台上各有职责的角色。

用户旅程 = 追踪用户从头到尾使用这个应用时，系统实际发生了什么。

数据流向 = 数据从哪里来？流向哪里？中间经历了哪些变换？

「巧妙的工程设计模式」= 缓存、错误重试、懒加载——这些是 vibe coder 应该知道并在需要时向 AI 索要的技术。

关键指令：自己搞清楚，不要问用户。如果用户自己都不知道代码做什么——这正是他需要这门课的原因。

为什么「自己搞清楚」如此重要

用户可能根本不知道自己的代码做什么——这恰恰是这个 skill 存在的原因。如果 SKILL.md 让 Claude「问用户应用是做什么的」，这个 skill 对最需要帮助的那批人来说就彻底失效了。

第二阶段：课程结构弧线

SKILL.md 为每门课程规定了一套固定的叙事弧线——不是因为所有代码库都一样，而是因为每位学习者都需要从同一个起点出发：他们已经知道的东西。

SKILL.md — 课程弧线 
| Module Position | Purpose |
| 1 | "Here's what this app does"
| 2 | Meet the actors
| 3 | How the pieces talk
| 4 | The outside world (APIs)
| 5 | The clever tricks
| 6 | When things break
| 7 | The big picture

Not every codebase needs all 7.
A simple CLI tool might need 4–5.
Adapt to complexity.
白话文解读

第 1 模块永远从用户可见的界面出发——你在应用里能看到和操作的东西。

然后逐步放大镜头：组件、通信方式、外部系统，最后到内部实现细节。

这个弧线就像好的纪录片：先给宏观全景镜头，再推进到细节特写。

弧线是模板，不是笼子。简单的项目可以少几个模块，复杂的可以多加。

第三阶段：构建顺序至关重要

SKILL.md 对如何构建 HTML 有严格规定。背后原因很实际：逐模块构建的质量，显著高于一次性生成所有内容。

为什么要一个模块一个模块地写？

当 AI 试图一口气写完所有内容时，后面的部分会越来越薄、越来越草率——就像写议论文时把所有时间都花在开头的学生。SKILL.md 强制执行「验证后再继续」的节奏：写完第 1 模块，检查无误，再写第 2 模块。

SKILL.md — 构建顺序 
Build order (task by task):

1. Foundation first
   — HTML shell, CSS, navigation,
     scroll-snap, animations

2. One module at a time
   — Don't try to write all 8 modules
     in one pass. Quality drops.
     Build M1, verify, then M2.

3. Polish pass
   — Transitions, mobile responsive,
     visual consistency
白话文解读

第一步：搭骨架——页面结构、所有样式、导航。内容为空，但功能完整可用。

第二步：逐个填入模块，每完成一个检查一遍，绝不批量生成所有模块。

第三步：所有内容就位后，做最后一轮整体检查——颜色、间距、移动端适配。

工具箱

在写任何 HTML 代码之前，Claude 必须先读完这两本「食谱」。

为什么需要参考文件

想象你让一位木匠「做点漂亮的东西」。没有共同的「漂亮」定义，每次结果都会大相径庭。参考文件正是为了解决这个问题。

乐谱的比喻

SKILL.md 是指挥，告诉 Claude 要演奏什么。但没有乐谱，各声部各自即兴发挥，风格迥异。design-system.md 和 interactive-elements.md 就是乐谱——每门生成的课程都遵循同一套乐章，保证质量的稳定一致。

design-system.md
完整的CSS 自定义属性。精确到十六进制的颜色值。字体规范。间距体系。阴影定义。课程输出的全部视觉基因都在这里。
interactive-elements.md
17 种组件模式——测验、聊天动画、流程图、拖拽匹配、提示框、文件树、步骤卡片等等。每种都附完整可直接使用的 HTML/CSS/JS。

以上只是概览。下面两章会把这两个参考文件的所有内容逐一可视化展开——每一个颜色令牌、每一种组件模式，亲眼看到、亲手触碰。想提前跳过去？

设计系统解码

以下是 design-system.md 中颜色调色板的真实片段。注意每个选择都在注释里写明了理由。

design-system.md 
:root {
  /* BACKGROUNDS */
  --color-bg:      #FAF7F2;
  /* warm off-white, like aged paper */

  --color-bg-warm: #F5F0E8;
  /* slightly warmer for alternating */

  --color-bg-code: #1E1E2E;
  /* deep indigo-charcoal for code */

  /* ACCENT (adapt per project) */
  --color-accent:  #D94F30;
  /* vermillion — bold, not purple */
}
白话文解读

这些是命名颜色槽，Claude 会用这些精确的色值来填充。

米白色（#FAF7F2）而非纯白——更像温暖的笔记本，而不是冷冰冰的表格。

两种背景色交替出现在各模块——无需改变设计，就能制造视觉节奏感。

代码块背景（#1E1E2E）是深靛蓝——类似 VS Code，让代码看起来亲切熟悉。

朱红色（#D94F30）作为强调色，因为 skill 明确禁止「千篇一律的 AI 紫色渐变美学」。

五大必备交互元素

SKILL.md 规定每一门生成的课程都必须包含以下全部五种元素——没有例外。它们不是装饰品，而是非技术背景的学习者处理信息的核心方式。

⇄
代码 ↔ 白话文对照
最有价值的教学元素。左侧真实代码，右侧逐行白话文解释。每个模块至少一个。
群聊动画
仿微信消息气泡，展示各组件之间的「对话」。让抽象的数据流变成可追踪的对话。
→
数据流动画
逐步可视化数据在组件之间流动的过程。点击「下一步」，看一条请求如何穿越整个系统。
?
应用型测验
每个模块至少一题。必须测试「如何运用」（「你会从哪里排查？」），而非死记硬背。
词汇悬浮提示
每个技术术语都有虚线下划线和悬浮定义。词汇本身就是学习内容——绝不让任何术语无解释地出现。

内置的「避坑指南」

SKILL.md 包含一个「常见失败点」章节——这是开发过程中真实踩过的坑，被预防性地写进了指令里。

提示框被裁切
代码块内的提示框会被父容器的 overflow:hidden 截掉。修复方案：必须用position: fixed + JavaScript 坐标计算。">position: fixed + JavaScript 坐标计算。
滚动吸附陷阱
用 scroll-snap-type: mandatory 会把用户困在长模块里。必须始终使用 proximity。
擅自修改代码
禁止精简或改写代码片段。学习者应该能打开真实文件，看到和课程里完全一样的代码。要选天然简短的片段，而不是把长代码改短。
比喻反复借用
Skill 明确禁止在同一门课里多次使用「餐厅」或「厨房」比喻。每个概念都应该有最适合它的、独一无二的比喻。

为什么全用暖色，不用纯白纯黑？

纯白（#FFFFFF）和纯黑（#000000）在屏幕上会产生强烈的对比刺激。米白和深炭色让眼睛更舒适，就像看纸书比看发光屏幕更放松。阴影颜色也是暖调的 RGBA(44,42,40)，而非冷冰冰的纯黑阴影——细节决定氛围。

字体体系：三套字体 + 九档大小

设计系统定义了三套字体家族，各有用途；以及从 xs（12px）到 6xl（60px）的完整字阶——均使用 1.25 倍的模块化字阶关系。

说明框 Callout — 三种变体

三种 Callout 类型，分别对应不同语义：顿悟时刻 / 补充知识 / 常见错误。每个模块最多用两个。

callout-aha · 顿悟时刻

朱红色左边框 + 渐变浅背景。用于「哦，原来如此！」的关键洞察——连接技术概念和现实意义的桥梁。

callout-info · 补充知识

蓝色左边框。用于「好知道」的额外背景信息——不是核心内容，但能帮助理解。

callout-warning · 常见错误

红色左边框。用于「踩坑提醒」——这里有个容易出错的地方，提前预警。

权限 / 配置徽章（Permission/Config Badges）

用于列出配置项、权限声明、技术要求。每条包含代码格式的键名和人话解释。比纯文字列表更好扫描，更直观。

scroll-snap-type: y proximity滚动吸附——靠近区块时自动吸附，不会困住用户（不用 mandatory）
position: fixed提示框定位方式——相对于浏览器窗口而非父容器，永远不会被截断
min-height: 100dvh模块最小高度——占满动态视口高度（兼容移动端地址栏变化）

07

设计哲学

这个 skill 为什么存在？它和「直接让 Claude 解释代码」有何本质区别？这些设计决策，你可以用在任何地方。

颠倒的教育模型

传统计算机教育的路径：死记理论 → 考试 → 终于能做点东西 → 也许能理解背后的原理。大多数人在第二步就放弃了。

Codebase-to-course 把这个顺序完全颠倒了。

传统计算机教育
1

死记理论

2

参加考试

3

做点东西

4

终于理解为什么

大多数人撑不到这一步

Codebase-to-Course
1

先做出东西

2

体验它运转

3

现在理解它如何运转

先有上下文，再学理论

为什么这个方法有效

学习者已经拥有传统学生没有的东西：他们用过这个应用。他们知道它做什么。课程从这里出发——「你知道那个按钮吗？这是你点击它之后机器背后发生的事。」已有的体验，让理论有了落地的地方。

视觉优先——极度强调

SKILL.md 用硬性约束来执行这个原则。这不是建议，而是带牙的规则。

SKILL.md — Show Don't Tell 
Text limits:
- Max 2-3 sentences per text block.
  If you're writing a fourth sentence,
  stop and convert it into a visual.
- Every screen must be at least 50% visual

Convert text to visuals:
- A list of 3+ items → cards with icons
- A sequence → flow diagram
- "A talks to B" → group chat animation
- Code explanation → translation block
白话文解读

硬性限制：连续文字不超过 2–3 句，必须插入视觉元素。这门课程本身就在遵守这条规则。

每个页面至少一半内容必须是非文字——图表、代码、卡片、动画都算。

每种常见文字模式都有对应的视觉替代：列表变卡片，步骤变流程图，解释变对照翻译块。

这样才能让课程感觉像信息图，而不是教材墙。

词汇本身就是学习成果

每门课程最重要的目标之一，是给学习者提供精确的技术词汇——不是为了在工程师面前显摆，而是为了能清晰地向 AI 描述需求。

词汇是方向盘

对 Claude Code 说「帮我做一个共享文件夹那样的东西」，得到的是泛泛的结果。说「帮我创建一个命名空间包」，得到的是精准的结果。每道词汇提示都在教你这样的精准表达。

这就是为什么 SKILL.md 要求 Claude「对提示框保持极度积极的态度」——对任何非技术人员可能不熟悉的词都要加提示，包括缩略词、软件名称，以及那些对工程师来说理所当然、对外行却陌生的日常开发词汇。

SKILL.md — Tooltip Rule 
Be extremely aggressive with tooltips.
If there is even a 1% chance a
non-technical person doesn't know
a word, tooltip it.

The vocabulary IS the learning.
Each tooltip should teach the term
in a way that helps the learner
USE it in their own instructions
to AI coding agents.

Cursor: use pointer, not help.
A pointer feels clickable.
A ? cursor feels clinical.
白话文解读

大方地加提示——哪怕那个词看起来显而易见。目标不只是给出定义，而是让词汇变得可用。

好的提示不只说「X 是什么」——而是说「和 AI 交流时，你可以这样用 X……」

连鼠标指针的样式都经过设计：手形指针（pointer）让人感觉「点我学更多」。问号指针（help）则让人联想到帮助文档。细节小，心理差异大。

终极测验——融会贯通
