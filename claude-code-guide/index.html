Chapter 01

它不是聊天机器人，
是一个「做事的循环」

你以为在聊天，其实在指挥一个流水线

大多数人第一次用 Claude Code，会像用微信聊天一样 -- 说一句，等它回一句。作者 Tw93 最初也是这样。但他很快发现，聊着聊着，Claude 就开始「犯糊涂」：说过的事情忘了，规则定了不遵守，工具越给越多却越来越乱。

折腾了一阵之后，他终于搞明白了一件事：Claude Code 不是一个「回答问题」的聊天工具，而是一个「做事情」的循环系统。

打个比方

想象你请了一个非常聪明的实习生来帮你。这个实习生的工作方式不是「你问我答」，而是一个不断重复的循环：

看看手头有什么信息 -> 动手做一步 -> 检查做得对不对 -> 没做完就回去再看再做

问题出在哪呢？如果你给实习生的「手头信息」是一堆杂乱的便签纸，或者根本没告诉它「做对了」长什么样 -- 再聪明的人也会迷路。

收集信息
->
动手做事
->
检查结果
->
完成 / 继续
核心认知

问题几乎从来不是「Claude 不够聪明」，而是你给了它错误的信息，或者做完了却没办法判断对不对。搞清楚这个循环，就找到了和 Claude 协作的钥匙。

这意味着什么？意味着你和 Claude 合作的质量，不取决于它有多聪明（它已经很聪明了），而取决于你有没有做好这五件事：

  1. 给它清楚的「上下文」-- 也就是做事需要知道的背景信息
  2. 给它合适的「工具」-- 也就是做事的手段
  3. 设好「控制」-- 也就是哪些事能做、哪些不能
  4. 要求「验证」-- 也就是怎么确认做对了
  5. 给予「记忆」-- 也就是跨越不同对话时，重要的事不被忘记
Chapter 02

你的书桌只有这么大

上下文管理 -- Claude Code 最重要的一课

这是整篇文章中 Tw93 反复强调的、最最最重要的一点。很多人觉得 Claude Code 有 200K 的上下文窗口，够大了。但实际情况是 --

打个比方

想象 Claude 的「上下文」就是一张固定大小的书桌。200K 听起来很大，但你一坐下来就会发现：

桌上已经摆好了一套办公文具（系统指令、工具定义等），这些是不能动的，一下子就占了 10-15% 的空间。

然后是你的工作手册（CLAUDE.md、规则文件等），又占了 5%。

真正留给你摊开文件、写东西、贴便签的空间，大概只有 80% 左右。

最要命的是：每接一个新的外部工具（MCP Server），就像在桌上多放一个工具箱 -- 5 个工具箱就能吃掉 12.5% 的桌面！

200K 总空间（你的书桌） | |-- 固定办公用品 (~15-20%) | |-- 系统指令（必备文具） | |-- 工具说明书（每个工具都要占位） <-- 隐形空间杀手！ | |-- 环境信息 | |-- 你的工作手册 (~5-10%) | |-- CLAUDE.md（合作契约） | |-- 记忆文件 | |-- 真正干活的空间 (~70-80%) |-- 你们的对话 |-- 读取的文件内容 |-- 做事的中间结果

但空间不够大还不是最大的问题。最大的问题是「太吵了」。

很多人把上下文当「容量问题」，但卡住的地方通常不是不够长，而是太吵了 -- 有用的信息被大量无关内容淹没了。 -- Tw93

就像你的书桌上堆满了各种文件、旧便签、用过的草稿纸，真正需要看的那页报告反而被埋在了底下。Claude 不是看不见，而是在一堆噪音里找不到重点了。

所以怎么办？像整理书桌一样整理信息

Tw93 给出了一个非常实用的「分层收纳」策略：

常备在桌上的

CLAUDE.md -- 你和 Claude 的合作契约。最核心的规则、怎么干活、绝对不能做的事。保持简短。

放在抽屉里按需拿的

Skills（技能文件）-- 特定任务的操作手册。平时不占桌面，需要做某类任务时才拿出来。

派人去别的房间查的

Subagents（子代理）-- 需要大量翻阅资料时，派一个助手去另一个房间查，只把结论带回来。

贴在门口的标语

Hooks（钩子）-- 不占书桌空间，但自动执行的规定。比如「每次改完代码自动检查拼写」。

黄金法则

偶尔用的东西，不要每次都摆在桌上。这一条看似简单，却是让 Claude 保持清醒、少犯错的最关键策略。

聊太久了怎么办？写交接单

和 Claude 聊久了（一两个小时），书桌上会堆满过程中的各种草稿，系统会自动「清理」-- 但清理的时候可能把重要的东西也扔了。Tw93 发现一个很实用的办法：

打个比方

就像下班前写一份交接单（HANDOFF.md）：现在做到哪了、试过什么方法、哪些走通了、哪些是死路、下一步该做什么。

第二天（开新对话）的时候，新的 Claude 只需要读这份交接单，就能无缝接手。比让系统自动「压缩回忆」靠谱得多。

另外还有几个实用的「书桌管理」命令：

  • /clear -- 清空桌面重新来过。当 Claude 被纠正了两次以上还搞不对，果断清桌子比继续在烂摊子上挣扎好
  • /compact -- 打包归档。把前面的对话压缩成摘要，腾出空间继续聊，但重点信息保留
  • /context -- 检查桌面。看看现在书桌上都摆了什么，哪些东西占了太多空间
Chapter 03

给它规矩、手册和助手

Skills、Hooks、Subagents 到底是什么

Claude Code 有一堆看起来很像的概念：Skills、Hooks、Subagents、Tools、MCP...... 光名字就能把人搞晕。Tw93 用了一句话说清楚了它们的区别：

给 Claude 新的动手能力，用 Tool / MCP。
给它一套做事方法，用 Skill。
需要隔离环境干活，用 Subagent。
要强制执行某些规定，用 Hook。 -- Tw93

这些东西乍一看很技术，但其实可以用一个公司的比喻来理解：

想象你是一个公司老板，Claude 是你雇的员工

CLAUDE.md（合作契约）= 员工入职时签的「员工手册」。里面写了公司的基本规定：怎么打卡、怎么报销、绝对不能做的事。每天上班都要遵守，所以要简洁明确。

Skills（技能）= 放在资料柜里的「操作手册」。比如「客户投诉处理流程」「新产品上架流程」。不需要时不用翻出来，需要时按手册一步步做。

Hooks（钩子）= 办公室里的「自动化规定」。比如门口的刷卡机（进门必须刷卡）、打印机旁的碎纸机标语（机密文件打完必碎）。不需要思考，不需要判断，触发了就自动执行。

Subagents（子代理）= 你派出去「跑腿」的助理。需要跑一趟图书馆查大量资料？派个助理去，只需要他回来告诉你结论就行，不用把整个图书馆搬到你办公室。

Tools / MCP（工具）= 给员工配的具体工具。计算器、扫描仪、邮件系统。每多一个工具，桌上就多占一块地方，所以别配太多用不到的。

关于 Skills 的三种类型

Tw93 在实际使用中总结出三种最实用的 Skill（操作手册）类型：

☑

检查清单型

就像飞行员起飞前的 Checklist -- 每一项都打勾才能走。比如「发布前检查」：代码能编译吗？版本号改了吗？更新日志写了吗？逐项过一遍，确保不漏。

⚙

工作流型

高风险操作的标准步骤，还自带「后悔药」。比如「数据迁移」：第一步备份、第二步预演、第三步正式执行、第四步验证。出了问题怎么回滚也写清楚了。

🔍

领域专家型

给 Claude 一个「诊断决策树」。出了问题先查什么、看到什么症状对应什么原因。就像医院的分诊手册 -- 不让它瞎猜，而是按路径排查。

Skill 的一个重要细节

每个 Skill 的「简介」会常驻在 Claude 的记忆里（占书桌空间），所以简介要写得短！Tw93 举例：一个 45 个词的简介可以优化成 9 个词，省了 80% 的空间。而且，用得少的 Skill（比如一个月用不到一次），干脆关掉自动加载，手动触发就好。

关于 Hooks 的核心思想

Hooks 最本质的价值是：把不能靠 Claude「记住」的事情，变成自动执行的流程。

不该用 Hook 做的
  • 需要阅读大量资料才能判断的事
  • 需要多步思考、权衡利弊的决策
  • 耗时很长的复杂流程
适合用 Hook 做的
  • 每次改完代码自动检查语法
  • 保护某些文件不被修改
  • 任务完成后自动发通知
  • 开始工作前自动加载环境信息

Tw93 算了一笔账：在一次包含 100 次代码修改的长对话中，每次自动检查省 30-60 秒，累积下来能省 1-2 小时。而且 -- 越早发现错误，越省时间。

关于 Subagents（子代理）的正确用法

打个比方

你在整理一份报告（主任务），突然需要查一堆统计数据。你有两个选择：

选择 A：自己去资料室，把 50 份报告搬到桌上一份份翻 -- 桌子马上就乱了，而且你原来在写的报告可能被埋掉。

选择 B：派一个助理去资料室查，只把查到的数字写在一张纸条上带回来 -- 你的桌子保持整洁，报告继续写。

Subagent 就是选择 B。它的核心价值不是「快」，而是「隔离」-- 不让大量的中间过程污染你的主书桌。

Subagent 常见错误
  • 给子助理和你一样的全部权限（失去了隔离意义）
  • 让子任务之间互相依赖、频繁传话
  • 不限制子助理的回答格式
Subagent 正确用法
  • 明确限制它能用什么工具
  • 规定好它的汇报格式
  • 设置时间/步数上限，防止跑偏
  • 让不同子任务各自独立运行
Chapter 04

先想清楚，再动手

Plan Mode、验证闭环、以及「怎么才算做完了」

Tw93 发现的一个重要教训是：急着让 Claude 出活，反而容易出问题。

打个比方

装修房子前，你会不会不画图纸就直接砸墙？大概率不会。Claude Code 有一个叫 Plan Mode（规划模式） 的功能，就是让 Claude 先「画图纸」：

在规划阶段，Claude 只看、只想、只提方案，不动手改任何东西。确认方案没问题了，再开始施工。

特别是对复杂的改动，这样做比「边改边想」靠谱得多，能大幅降低在错误方向上越走越远的概率。

如果你说不清「什么叫做完」，就别丢给 Claude

这可能是整篇文章里最具普遍意义的一句话：

假如一个任务你都说不清楚「什么叫做完了」，那它大概率也不适合直接扔给 Claude 自主完成。验证标准本身都没有，Claude 再聪明也跑不出正确答案。 -- Tw93

这不只适用于 Claude Code，也适用于你和任何 AI 工具的合作。给任务的时候，一定要同时告诉它「怎么检查做对了」：

  1. 做完之后跑哪些检查？（比如：代码能编译通过、测试全部通过）
  2. 如果失败了，先查什么？
  3. 成功的样子具体长什么样？（比如：页面上能看到 XX、日志里出现 YY）
验证的三个层级

最基础：自动检查 -- 语法对不对、编译能不能过、自动测试通不通过。
中间层：功能验证 -- 页面效果对不对、流程走得通不通。
最高层：人工确认 -- 上线后数据正不正常、用户反馈怎么样。

哪怕你只做到第一层，也比完全不验证强太多了。

Chapter 05

你的协作契约：CLAUDE.md

写给 Claude 看的「合伙人协议」，越短越好

CLAUDE.md 是 Claude Code 里最核心的配置文件。但 Tw93 强调：它不是文档，不是知识库，而是一份每次合作都要遵守的「契约」。

打个比方

想象你和一个合伙人签了一份协议。好的协议是什么样的？短、清楚、每条都可执行。

坏的协议呢？「我们要做一家伟大的公司、要写高质量的代码、要注重用户体验」-- 这些话说了等于没说。

好的协议是：「每周五下午 3 点开例会」「所有支出超过 500 元需要审批」「绝对不碰客户的个人数据」。

不该放进去的
  • 大段背景介绍
  • 完整的技术文档
  • 空泛的原则（如「写高质量代码」）
  • Claude 自己看代码就能推断的信息
  • 低频使用的专业知识（放 Skills 里）
应该放进去的
  • 怎么编译、怎么测试、怎么运行
  • 关键的项目结构和模块边界
  • 代码风格和命名规则
  • 绝对不能做的事（NEVER 列表）
  • 压缩记忆时必须保留什么

Tw93 分享了一个他最喜欢的技巧：每次你纠正了 Claude 的一个错误，就让它自己更新 CLAUDE.md --

你：「这里不应该用 XX 方式，应该用 YY 方式。」 Claude：「明白了，已修改。」 你：「把这条加到你的 CLAUDE.md 里，下次别再犯了。」 效果：Claude 会自己给自己补规则，用久了确实越来越少犯同样的错。

但也要注意：定期回头看看，时间久了总有些规则过时了。当初管用的限制，现在未必还适合。

起步建议

一开始甚至可以什么都不写。先用起来，等你发现自己反复跟 Claude 说同一件事的时候，再把它补进去。从需求出发，而不是从模板出发。

Chapter 06

三个成长阶段

从聊天到协作到系统设计

Tw93 认为，使用 Claude Code 大概会经历三个阶段：

🗪

第一阶段：聊天机器人

把 Claude 当聊天工具用，问一句答一句。偶尔让它帮忙写段代码、改个 Bug。效果时好时坏，觉得 AI 也就那样。

🔧

第二阶段：工具使用者

开始理解各种功能 -- Skills、Hooks、Subagents，开始配置 CLAUDE.md，开始管理上下文。效果明显提升，但还在不断摸索边界。

🏗

第三阶段：系统设计者

关注点从「这个功能怎么用」变成「怎么让 AI 在约束下自己跑起来」。你不再只是使用 Claude，而是在设计一个人机协作系统。这和第一阶段，感觉差很多。

送给你的实用锦囊

最后，从这篇文章里提炼出来最适合非技术背景用户立刻用起来的建议：

对话不要拖太久

感觉 Claude 开始犯迷糊了？被纠正两次还是搞不对？果断开个新对话。新的上下文比反复调整有用。

说清楚「怎么算完成」

布置任务时，同时告诉 Claude 怎么检验结果。没有验收标准的任务，Claude 再聪明也只能靠猜。

让它先想再做

复杂任务用 Plan Mode。让 Claude 先出方案不动手，确认OK了再执行。就像装修先看图纸。

少即是多

不要疯狂加工具和规则。每多加一样，Claude 的「书桌」就少一块空间。用不到的就关掉。

让 Claude 自己学

每次纠正错误后，让它把教训写进 CLAUDE.md。时间久了，它犯同样错误的概率会越来越低。

写交接单

长任务中途需要换对话？让 Claude 先写一份「当前进度总结」，下个对话接着来，不丢上下文。

🛠 -> 📋 -> ✅ -> 🚀
理解系统 -> 整理信息 -> 建立验证 -> 高效协作
这就是从 Tw93 半年踩坑中提炼出的核心路径
这些是半年折腾下来的一些总结，肯定还有很多没有挖掘到的地方。如果大伙有用得更溜的技巧，欢迎告诉我。 -- Tw93，原文结语
